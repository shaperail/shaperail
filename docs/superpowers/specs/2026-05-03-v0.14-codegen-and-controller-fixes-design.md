# v0.14.0 — Codegen and Controller Fixes (Design)

**Date:** 2026-05-03
**Status:** Approved
**Target release:** `v0.14.0` (single bundled `feat!:` minor bump)

## Background

Four bug reports filed against `shaperail-{core,codegen,runtime} 0.13.1` describe defects that, taken together, indicate a coherence problem at two seams:

1. The codegen-to-DB seam: SQL says `NOT NULL DEFAULT`, the Rust struct says `Option<T>`.
2. The YAML-to-Rust controller seam: too small a surface — single hook only, no path params, generated modules unreachable from tests.

This spec resolves all four in one release.

## Decisions

| ID | Decision |
|---|---|
| D1 | Ship as a single bundled `feat!:` PR producing `0.14.0`. |
| D2 | The codegen Option rule honors **all** column-level non-null sources: `required`, `default`, `generated`, `primary`. `nullable: true` always wins. |
| D3 | `before:` / `after:` accept `String | Vec<String>` polymorphically; same key, untagged enum (`HookList`). Empty arrays rejected by validator. Apply symmetrically to `after:`. |
| D4 | Add `ctx.path_params: HashMap<String, String>` plus a typed convenience `ctx.path_param(name) -> Option<&str>`. Do NOT auto-inject `id` into `ctx.input`. |
| D5 | Codegen emits `pub mod <resource>_controller;` and a `#[doc(hidden)] pub mod resources { ... }` aggregator in `generated/mod.rs`. Document the `pub use generated::resources;` one-liner for `lib.rs`; scaffold it in `shaperail init`. |

## Section 1 — Issue 1: Output type honors DB nullability

### Rule (replaces `model_field_is_optional`)

Pseudocode:

```
field is Option<T> in the codegen struct
    iff
nullable = true
    OR
(NOT primary AND NOT required AND default is None AND generated = false)
```

`nullable: true` always wins, even with `default:` set — the column genuinely admits NULL.

### Behavior matrix

| YAML | DB column | Codegen type |
|---|---|---|
| `{ type: integer, required: true }` | `BIGINT NOT NULL` | `i64` |
| `{ type: integer, required: true, default: 0 }` | `BIGINT NOT NULL DEFAULT 0` | `i64` |
| `{ type: integer, default: 0 }` *(was bug)* | `BIGINT NOT NULL DEFAULT 0` | **`i64`** |
| `{ type: timestamp, generated: true }` *(was bug)* | `TIMESTAMP NOT NULL DEFAULT NOW()` | **`DateTime<Utc>`** |
| `{ type: string, nullable: true }` | `VARCHAR NULL` | `Option<String>` |
| `{ type: string, nullable: true, default: "x" }` | `VARCHAR NULL DEFAULT 'x'` | `Option<String>` |
| `{ type: uuid, primary: true, generated: true }` | `UUID PRIMARY KEY DEFAULT gen_random_uuid()` | `Uuid` |

### Affected code

- `shaperail-codegen/src/rust.rs` — `model_field_is_optional()` (~line 1324) and `field_is_required()` (~line 1328) updated to the rule above. The two functions become exact inverses.
- `value_expression` / `generated_value_expression` paths — `Some(...)` wrapping conditions follow the new predicate.
- `shaperail-codegen/tests/snapshot_tests.rs` — golden files regenerated; that diff is the canonical proof of the fix.
- Examples (`examples/incident-platform`, `examples/multi-tenant`) regenerate. Any non-codegen Rust code in those examples that touched the changed fields is updated.

### Input-body invariant

CREATE/UPDATE input payloads remain unchanged. A field with `default:` is still optional in the request body — the caller may omit it, and the DB fills the default. Only the SELECT-side `pub struct` flips type. This matches the bug-report's framing: "default: implies required: true for the output type, not the input type."

## Section 2 — Issue 2: Hook chains under `before:` and `after:`

### Type change

`shaperail-core/src/endpoint.rs`:

```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(untagged)]
pub enum HookList {
    Single(String),
    Multi(Vec<String>),
}

impl HookList {
    pub fn names(&self) -> &[String] {
        match self {
            HookList::Single(name) => std::slice::from_ref(name),
            HookList::Multi(names) => names.as_slice(),
        }
    }
}

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(deny_unknown_fields)]
pub struct ControllerSpec {
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub before: Option<HookList>,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub after: Option<HookList>,
}
```

### YAML grammar

```yaml
controller: { before: validate_org }                          # scalar form
controller: { before: [validate_currencies, validate_org] }   # array form
controller: { before: validate_x, after: [enrich_a, enrich_b] }
```

### Run semantics

- Hooks run in declaration order, sequentially, on the same `Context`.
- First `Err(_)` short-circuits the chain; remaining hooks do not run.
- WASM hooks (entries starting with `wasm:`) may be freely mixed with Rust hook names.
- An empty array (`before: []`) is a validator error: SR063 ("controller hook list must not be empty"). (SR065 is already taken; SR063 is the next free code in the controller-validation cluster.)

### Affected code

- `shaperail-core/src/endpoint.rs` — `ControllerSpec`, plus updated `has_wasm_before` / `wasm_before_path` (and `_after_` siblings) to scan a `HookList`.
- `shaperail-runtime/src/handlers/crud.rs:741,803` and `routes.rs:212` — replace `c.before.as_deref()` with iteration over `c.before.as_ref().map(|h| h.names()).unwrap_or(&[])`.
- `shaperail-codegen/src/{validator,diagnostics,openapi,rust}.rs` — every site that currently reads `controller.before` as a single string iterates instead.
- New validator rule SR063 (empty hook list).
- `dispatch_controller`'s signature does not change; the loop lives at the call site.

## Section 3 — Issue 3: Path params on `ControllerContext`

### API change

`shaperail-runtime/src/handlers/controller.rs`:

```rust
pub struct Context {
    /* existing fields unchanged: input, data, user, pool, headers,
       response_headers, tenant_id, session, response_extras */
    pub path_params: std::collections::HashMap<String, String>,
}

impl Context {
    pub fn path_param(&self, name: &str) -> Option<&str> {
        self.path_params.get(name).map(String::as_str)
    }
}
```

### Population

The runtime populates `path_params` before invoking any controller. For the four built-in CRUD endpoints:

| Endpoint | `path_params` |
|---|---|
| `list` (`GET /resource`) | `{}` |
| `get` (`GET /resource/:id`) | `{"id": "<uuid-or-key>"}` |
| `create` (`POST /resource`) | `{}` |
| `update` (`PATCH /resource/:id`) | `{"id": "<...>"}` |
| `delete` (`DELETE /resource/:id`) | `{"id": "<...>"}` |

For custom endpoints with arbitrary path templates, every `:name` segment becomes an entry.

### Non-decision

`ctx.input` is **not** auto-populated with `id` from the path. Authors who want id-in-input write `ctx.input.insert("id", json!(ctx.path_param("id")))` themselves. Reason: the existing `INSERT (...input...)` codegen path would otherwise need to special-case the `id` column, and the explicit single-line read at the top of an update hook is honest about where the value came from.

### Affected code

- `shaperail-runtime/src/handlers/controller.rs` — struct field + helper.
- `shaperail-runtime/src/handlers/crud.rs` — populate from `actix_web::web::Path` before dispatching the before-hook chain.
- `shaperail-runtime/src/handlers/routes.rs` — same population for custom-handler dispatch.
- All `Context { ... }` literal sites in the test suite get the new field.
- A `Default for Context`-style or `Context::test()` helper in `shaperail-runtime/src/test_support.rs` updated so tests don't need to know the exact field set.

## Section 4 — Issue 4: Public controller modules

### Codegen output

`generated/mod.rs` becomes:

```rust
// Per-resource modules — pub so the aggregator can re-export them.
// Existing intra-crate imports (e.g. `use super::users_controller`) keep working.
pub mod users_controller;
pub mod orders_controller;
// …

/// Re-exports of every controller module under one path so integration tests
/// can reach `pub` helpers via `crate::resources::<name>_controller::*`.
///
/// `#[doc(hidden)]` keeps this off the docs.rs surface.
#[doc(hidden)]
pub mod resources {
    pub use super::users_controller;
    pub use super::orders_controller;
    // …
}
```

### `lib.rs` expectation

Library projects (which expose helpers to integration tests) add one line to `lib.rs`:

```rust
mod generated;
pub use generated::resources;
```

`shaperail init`'s template scaffolds this line.

### Affected code

- `shaperail-codegen/src/rust.rs` — wherever `mod {resource}_controller;` is currently emitted, change to `pub mod`. After all per-resource modules, emit the `#[doc(hidden)] pub mod resources { ... }` block.
- `shaperail-cli/src/commands/init.rs` (or the init template, wherever it lives) — update the scaffolded `lib.rs` to include `pub use generated::resources;`.
- `agent_docs/architecture.md` and `docs/testing.md` — document the import path test authors should reach for: `use my_app::resources::users_controller::*;`.

## Section 5 — Migration

### Breaking changes (`CHANGELOG.md` Breaking section)

1. Struct fields generated for columns with `default:` or `generated:` flip from `Option<T>` to `T`. Mechanical fix: remove `.unwrap_or(...)`, `.unwrap()`, or `match Some(x)` patterns.
2. `ControllerSpec.before` / `after` change from `Option<String>` to `Option<HookList>`. Affects anyone reading the parsed AST in custom tooling — rare in practice.

### Additive

3. `controller.before` / `controller.after` accept arrays in YAML.
4. `ctx.path_params` map and `ctx.path_param(name)` helper available in all controllers.
5. `generated/mod.rs` exposes `#[doc(hidden)] pub mod resources` so integration tests can reach controller helpers.

### Docs to update (CLAUDE.md mirror requirement)

| `agent_docs/` | `docs/` |
|---|---|
| `codegen-patterns.md` — Option rule + matrix | `resource-guide.md` — inline note on default/generated → non-Option |
| `hooks-system.md` — chain semantics, ordering, short-circuit, path params | `controllers.md` — public mirror |
| `resource-format.md` — `before:` / `after:` accept arrays | `resource-guide.md` — public mirror |
| `architecture.md` — public controller path | `testing.md` — how integration tests import helpers |

`CHANGELOG.md` `[Unreleased]` section names every change. `CLAUDE.md` does not need a process update — no codebase convention shifts.

## Section 6 — Testing strategy

- **Codegen unit tests:** `model_field_is_optional` exhaustive matrix covering all six rows in Section 1's table, plus `HookList` deserialization tests for `Single`, `Multi`, and empty-array-rejection-via-validator.
- **Codegen snapshot tests:** `shaperail-codegen/tests/snapshot_tests.rs` golden files regenerated. The diff is the canonical proof of Section 1's fix.
- **Validator tests:** SR063 fires on `before: []` and `after: []`. Existing empty-string check (`controller.before name`) keeps working for the scalar form.
- **Runtime integration tests** (`shaperail-runtime/tests/handler_tests.rs`):
  - hook chain runs in declaration order;
  - first `Err(_)` short-circuits subsequent hooks;
  - `ctx.path_params["id"]` populated for PATCH `/users/:id` before-hook;
  - WASM and Rust hooks coexist in the same chain (skipped when `wasm-plugins` feature is off).
- **End-to-end smoke:** `examples/incident-platform` regenerates and compiles; one integration test reaches `crate::resources::<x>_controller::*`.

The `/run-checks` gate (`cargo build --workspace`, `cargo test --workspace`, `cargo clippy -- -D warnings`, `cargo fmt --check`) is the final gate.

## Out of scope

- Wider refactor of `ControllerContext` (e.g. exposing `&HttpRequest`). Path params solve the canonical UPDATE-by-id case; broader actix exposure is a separate design.
- Changing the controller signature to be sync or fallible-without-Result. The existing `&mut Context -> Result<(), ShaperailError>` shape stays.
- Backfilling `default:`/`generated:` rules for already-published projects beyond the compile-error guidance — the bump is honest, the fix is mechanical, and we don't ship a codemod.
