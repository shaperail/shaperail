# Batch 4 — Docs & `test_support`

**Date:** 2026-05-02
**Issues:** #4 (no `test_support` for in-process server spawning), #10 (JWT Claims shape undocumented)
**Risk:** Low. Purely additive runtime API + a CLI template tweak. No breaking changes.

## Goal

Make integration testing a Shaperail app as cheap as the zero2prod `TestApp` pattern, and stop forcing consumers to read the runtime source to figure out the JWT claim shape.

## Non-Goals

- Refactoring the entire bootstrap (`HttpServer::new(...)` glue) out of the user's generated `main.rs` into `shaperail-runtime::server::build_server`. That is a separate, larger initiative tracked as a follow-up.
- Adding a full test-fixture/factory framework. We give the bare minimum: spawn, migrate-once, expose addr.
- Changing the JWT validation logic. Only adds re-exports and improved log lines.

## Change 1 — `shaperail_runtime::test_support` (#4)

### 1a — New module

**File:** `shaperail-runtime/src/test_support.rs` (new), exported from `lib.rs` behind `#[cfg(any(test, feature = "test-support"))]`.

Add an opt-in cargo feature so production builds do not pay for the helper:

```toml
# shaperail-runtime/Cargo.toml
[features]
test-support = []
```

### 1b — Public surface

```rust
pub struct TestServer {
    addr: SocketAddr,
    handle: tokio::task::JoinHandle<std::io::Result<()>>,
}

impl TestServer {
    pub fn address(&self) -> SocketAddr { self.addr }
    pub fn port(&self) -> u16 { self.addr.port() }
    pub fn url(&self, path: &str) -> String {
        format!("http://{}{}", self.addr, path)
    }
    pub async fn shutdown(self) -> std::io::Result<()> { /* abort handle */ }
}

impl Drop for TestServer {
    fn drop(&mut self) { self.handle.abort(); }
}

pub async fn spawn_with_listener<F, S>(
    listener: std::net::TcpListener,
    factory: F,
) -> std::io::Result<TestServer>
where
    F: FnOnce(std::net::TcpListener) -> std::io::Result<S> + Send + 'static,
    S: actix_web::dev::Server + 'static,
```

The `factory` closure is the consumer's `build_server(listener)` from their `lib.rs`. `spawn_with_listener` only does:

1. Set the listener non-blocking.
2. Call the factory to build the `actix_web::dev::Server`.
3. `tokio::spawn` the server future, capturing the join handle.
4. Return `TestServer` with the bound address read from the listener *before* it was consumed (capture `local_addr()` first).

### 1c — Migration gating helper

```rust
pub async fn ensure_migrations_run(pool: &sqlx::PgPool) -> Result<(), sqlx::Error>;
```

Internal `tokio::sync::OnceCell<()>` gates a single `sqlx::migrate!()` invocation across all tests in the process. Subsequent calls are O(1) — no advisory-lock round trip. This sidesteps the parallel-test thrash described in the issue.

The helper does **not** create or drop databases; that responsibility stays with the consumer's test harness (`testcontainers`, manual setup, etc.). We are only solving the "every test thread takes the migration lock" problem.

### 1d — `shaperail init` template update

**File:** `shaperail-cli/src/commands/init.rs`

Currently the template emits a single `src/main.rs`. Change to emit:

- `src/lib.rs` — exposes `pub fn build_server(listener: TcpListener) -> std::io::Result<Server>` containing the existing bootstrap that lives at `init.rs:1500–1750`.
- `src/main.rs` — a thin binary:
  ```rust
  use std::net::TcpListener;
  fn main() -> std::io::Result<()> {
      let listener = TcpListener::bind(("0.0.0.0", config_port()))?;
      tokio::runtime::Runtime::new()?.block_on(<crate>::build_server(listener)?)
  }
  ```
- `Cargo.toml` — add `[lib]` target alongside `[[bin]]`.
- `tests/integration.rs` (new, generated by `shaperail init`) — a single smoke test that calls `spawn_with_listener` and hits `/health`. Demonstrates the canonical pattern.

This makes every new project test-ready out of the box.

### 1e — `docs.rs` example

Add a top-of-module rustdoc example to `test_support.rs`:

```rust
//! ```no_run
//! # use std::net::TcpListener;
//! # async fn run() -> std::io::Result<()> {

```

## Change 2 — JWT Claims documentation (#10)

### 2a — Re-export `Claims`

**File:** `shaperail-runtime/src/auth/mod.rs`

Add line: `pub use jwt::Claims;` (currently only `JwtConfig` is re-exported).

### 2b — Document required-claim shape on `Claims`

**File:** `shaperail-runtime/src/auth/jwt.rs`

Expand the doc comment on `pub struct Claims` (line 5) to spell out:

- `sub`: user ID (UUID string).
- `role`: must match a role declared in the resource YAML's `auth:` lists.
- `iat`/`exp`: unix seconds.
- `token_type`: must be `"access"` to authorize a request — `"refresh"` is only valid against the refresh endpoint.
- `tenant_id`: required for every non-`super_admin` role accessing a tenant-scoped resource (M18). Missing or null on tenant-scoped requests → 401.

Include a short rustdoc example showing how to mint a token for tests using `JwtConfig::encode_access_with_tenant`.

### 2c — Improve auth middleware error logging

**File:** `shaperail-runtime/src/auth/extractor.rs`

At line 60, currently:

```rust
if claims.token_type != "access" {
    return Err(ShaperailError::Unauthorized);
}
```

Add a `tracing::warn!` line with structured fields *before* the early return:

```rust
if claims.token_type != "access" {
    tracing::warn!(
        token_type = %claims.token_type,
        sub = %claims.sub,
        "JWT rejected: token_type must be \"access\""
    );
    return Err(ShaperailError::Unauthorized);
}
```

Same pattern for the JWT decode failure on line 59 (log `error = %e` once, before the 401).

This turns a silent 401 into something a developer can actually grep for in dev logs.

### 2d — Document the test-token recipe in `agent_docs/`

**File:** `agent_docs/testing-strategy.md` (or a new `agent_docs/auth-claims.md` if the testing doc is already busy).

Add a "Minting a test JWT" section that imports `Claims` and calls `JwtConfig::new(secret, ttl, ttl).encode_access_with_tenant(...)`. Pure documentation; no code change beyond what 2a/2b already give us.

## Acceptance Criteria

1. `cargo add shaperail-runtime --features test-support` exposes `shaperail_runtime::test_support::{spawn_with_listener, ensure_migrations_run, TestServer}`.
2. A fresh `shaperail init demo` produces a project where `cargo test` runs the generated `tests/integration.rs` and it passes against a running Postgres.
3. `cargo doc -p shaperail-runtime --features test-support --open` shows the `test_support` module with a working `no_run` example.
4. `shaperail_runtime::auth::Claims` is re-exported and shows up in `cargo doc`.
5. Hitting an auth-protected endpoint with `token_type: "refresh"` produces a `tracing` warn line including the rejected `token_type` value.
6. `cargo build --workspace`, `cargo test --workspace --all-features`, and `cargo clippy --workspace --all-features -- -D warnings` all pass.

## Rollout

Single PR. Non-breaking — existing user projects continue to work because we only changed the *generated* template and added new opt-in API surface. New init runs pick up the lib/bin split automatically.
